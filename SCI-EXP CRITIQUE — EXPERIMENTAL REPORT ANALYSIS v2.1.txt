🧪[SCI-EXP CRITIQUE — EXPERIMENTAL REPORT ANALYSIS v2.1]

This critique is written as a single continuous piece of analysis, not a checklist.
Each step must open from the conclusion of the previous one.
Read the report in full before executing. Write in English.
Do not summarize. Interrogate.
Do not infer missing artifacts. If an artifact is not present or not verifiable, treat it as absent.

─────────────────────────────────────────────────────
PRECHECK — ARTIFACT SUFFICIENCY GATE
─────────────────────────────────────────────────────
Execute this before STEP 0.

First, determine whether the report contains enough material to support a valid critique.
Check for the presence of:
- report body
- explicit verdict-bearing figures
- source anchors (source_path and/or equivalent)
- sha256 and/or hash manifest when traceability is claimed
- deviation log when HOLD / rerun / path change language appears
- prior-cycle reference material when cross-cycle comparison is made

Declare exactly one mode:

FULL       — Enough evidence exists to evaluate the primary claim
             and execute STEP 0 through STEP 5 normally.

PARTIAL    — Enough evidence exists to evaluate the primary claim,
             but one or more secondary traceability elements are missing.
             Continue through STEP 0 through STEP 5.
             Mark any conclusion that depends on the missing secondary
             artifact as partially traceable.

LIMITED    — One or more artifacts required to evaluate the primary claim
             are missing, but at least one central claim remains evaluable.
             Continue only in constrained form.
             Mark any conclusion that depends on the missing required
             artifact as not traceable.

BLOCKED    — The report does not contain enough material
             to support a valid critique.

PRECHECK output must be exactly 2 lines:
Line 1: PRECHECK MODE: FULL / PARTIAL / LIMITED / BLOCKED
Line 2: MISSING ARTIFACTS: none / [explicit list]

If mode = BLOCKED:
Stop after PRECHECK.
Name the single blocking absence.
Do not perform STEP 0 through STEP 5.

If mode = PARTIAL or LIMITED:
Continue, but do not repair missing evidence with inference.

Use only these traceability terms in the critique body:
- traceable
- partially traceable
- not traceable

Do not substitute weaker language such as "unclear," "suggestive," or "weakly supported"
when the real issue is traceability class.

─────────────────────────────────────────────────────
EVIDENCE PRECEDENCE
─────────────────────────────────────────────────────
When artifacts conflict, resolve them in this order:
1. Measured artifact or raw result file
2. Hash manifest / trace log / deviation log
3. Inline figure or table in the report
4. Narrative interpretation in the report
5. Cross-cycle comparison prose

A higher-ranked artifact overrides a lower-ranked one.
If two artifacts at the same rank conflict, name the conflict explicitly
and do not resolve it by guesswork.

─────────────────────────────────────────────────────
STEP 0 — EXPERIMENT CLASS (2 lines max)
─────────────────────────────────────────────────────
Classify the experiment as one of the following.
State the deciding reason in one sentence.

  PARITY      — Baseline reproduction. Goal is identity verification
                against a prior cycle.
  EXTENSION   — Adding a new component or signal on top of a
                validated baseline.
  RCA         — Failure root-cause analysis and patch verification.
  ABLATION    — Measuring the effect of turning one component on/off
                under otherwise identical conditions.
  MULTIAXIS   — Quantitative variance and governance delta comparison
                across multiple prior cycles.

If the experiment is composite, declare PRIMARY and SECONDARY.
Example: PRIMARY=EXTENSION, SECONDARY=PARITY

─────────────────────────────────────────────────────
STEP 1 — CLAIM INTEGRITY (40% of critique)
─────────────────────────────────────────────────────
Examine whether the report’s central claims are actually supported by the measured data.
Do not summarize. Interrogate.

Work through the following four checks in order.
Do not break the flow between them.

  1.1 CENTRAL CLAIM
      What is the report trying to prove?
      Does that match what was actually measured?
      If there is a gap between the stated objective and the
      measured evidence, name it precisely.
      A claim that is technically true but measured on a different
      path than the one stated counts as a gap.

  1.2 SCOPE BOUNDARY
      Does any part of the report cross the
      methodology / computational stability / reproducibility scope lock?
      Systemic causality, definitive physical yield assertions, and direct
      real-world application recommendations appear not only in explicit
      sentences but also in interpretive framing — catch both.
      A phrase that implies a direct improvement in physical or
      phenomenological outcomes based solely on computational or
      intermediate metrics is still a boundary violation.

  1.3 VERDICT LEGITIMACY
      Is each PASS / BLOCK / HOLD judgment traceable to
      a specific measured value?
      Composite verdict labels such as PASS_NON_DEGRADATION
      must have each component independently verified.
      If a verdict is constructed through narrative rather than
      derived from measurement, identify the exact sentence
      where that construction occurs.

  1.4 HOLD HANDLING
      If a HOLD event occurred:
      Is the cause stated explicitly?
      Is the disposition of the HOLD documented
      (isolated / patched / carried forward)?
      If the HOLD is described as "not a failure,"
      is that characterization supported by data
      or is it editorial framing?
      A HOLD that is carried forward without a stated cause
      is an open liability for the next cycle.

Close STEP 1 by naming the single most vulnerable claim
in one sentence.
That sentence must open STEP 2.

─────────────────────────────────────────────────────
STEP 2 — TRACEABILITY AUDIT (30% of critique)
─────────────────────────────────────────────────────
Starting from the vulnerable claim named at the end of STEP 1,
audit whether the report’s key figures are fully reconstructable.

  2.1 ARTIFACT CHAIN
      Does every key figure carry a source_path and sha256?
      A sha256 with no confirmed file existence is incomplete.
      Does the hash manifest agree with the sha256 values
      cited inline in the report?
      Name any figure that is cited without both anchors.
      If the report claims traceability while the anchor is missing,
      state that the claim does not hold.

  2.2 DEVIATION LOG COMPLETENESS
      Were all deviation events — HOLDs, reruns, path changes —
      recorded in the deviation log?
      For each logged event: is the cause stated,
      or only the outcome?
      Name any event that appears in the report narrative
      but is absent from or incomplete in the deviation log.

  2.3 INTERPRETATION GAP
      Distinguish two failure modes and name instances of each:
      — Figure present, interpretation absent.
      — Interpretation present, figure absent.
      Audit trace fields in particular — deterministic constraint flags,
      invariant violation counts, strict_recheck_status —
      are common sites for the first failure mode.
      They appear in reports as data but receive no
      stated meaning.

  2.4 CROSS-CYCLE CONSISTENCY
      When figures from prior cycles (EXP-032, EXP-033, etc.)
      are cited for comparison, confirm that the comparison
      basis is identical: same anchor, same dataset, same arm.
      A comparison made across different anchors is invalid
      regardless of whether the numbers look stable.
      Name any comparison that mixes baselines.

─────────────────────────────────────────────────────
STEP 3 — SERIES CONTINUITY (20% of critique)
─────────────────────────────────────────────────────
From the traceability gaps above, determine whether this report occupies the correct position in the experimental series.
This is not about the report in isolation — it is about whether the series holds together.

If no prior cycle exists, explicitly declare NOT APPLICABLE for cross-cycle continuity
and evaluate only whether the report states its own forward handoff correctly.
Do not fabricate series debt where no series baseline exists.
Omissions in a non-series report do not create series debt.

  3.1 HANDOFF CONTRACT
      What open questions did the prior experiment leave?
      Does this report resolve them, carry them forward,
      or ignore them?
      If a prior open question is ignored, determine whether
      that was a deliberate scoping decision or an omission.
      Omissions create silent debt in the next cycle.

  3.2 NEXT CYCLE READINESS
      Is the handoff scope for the next experiment stated?
      A handoff that says only "carry the RCA forward"
      is not a handoff — it is a deferral.
      A valid handoff names the opening question,
      names what is out of scope, and names what
      the next cycle must not do.
      Evaluate whether this report meets that standard.

  3.3 NARRATIVE DRIFT
      The series operates under one governing frame:
      methodological and governance integrity over performance improvement.
      Does this report hold that frame?
      Any sentence that reads stability metric movement
      as evidence of systemic superiority is a drift event.
      Name it if it occurs.
      Quote the exact sentence when possible.

─────────────────────────────────────────────────────
STEP 4 — PUBLICATION READINESS (10% of critique)
─────────────────────────────────────────────────────
From the continuity judgment above, determine whether this report can serve as the source of truth
for downstream public writing — dev.to, Substack, Medium, or equivalent.

  4.1 INTERNAL RULE COMPLIANCE
      If the report contains downstream writing rules,
      check whether the report itself violates them.
      A report that prohibits hash-free figures
      while citing hash-free figures is self-contradicting.

  4.2 MISREAD RISK
      Identify the single point in the report where
      an external reader is most likely to draw
      a conclusion the report does not intend.
      State the misread direction precisely.
      Do not generalize.
      Name the sentence or figure
      and the specific wrong conclusion it invites.

─────────────────────────────────────────────────────
STEP 5 — PRIORITY FIX
─────────────────────────────────────────────────────
Synthesize STEP 1 through STEP 4.

Name exactly one thing to fix first.
Explain why it takes priority over everything else
in exactly 2 sentences.

Do not produce a list.
Do not rank second or third priorities.
Everything else carries forward to the next round.

OPTIONAL — NEXT LIABILITY
Name one issue that is not the priority fix
but is likely to surface immediately after the priority fix is implemented.
Use exactly 1 sentence.
Do not rank it against the priority fix.

─────────────────────────────────────────────────────
OPERATIONAL RULES
─────────────────────────────────────────────────────
Do not break flow between steps.
Each step must open from the conclusion of the previous one.
When citing figures, use the report’s exact notation.
Do not paraphrase verdict labels — PASS, BLOCK, HOLD
carry specific operational meaning in this series.
Replace "good / bad" with "holds / does not hold."
Replace "clear / unclear" with traceability class:
traceable / partially traceable / not traceable.
Treat an unverified artifact as absent.
Do not reconstruct missing baselines.
Do not assume file existence from a hash alone.
Do not assume causal interpretation from intermediate metrics alone.
If the report overreaches, name the exact sentence that overreaches.

─────────────────────────────────────────────────────
ROUND PROTOCOL
─────────────────────────────────────────────────────
Round 1 — Execute PRECHECK, then STEP 0 through STEP 5.
           Use OPTIONAL — NEXT LIABILITY only if it materially helps the next round.

Round 2+ — Begin by confirming whether the STEP 5 fix
           from the prior round was implemented.
           Then re-execute STEP 1 through STEP 5 only,
           focusing on:
           - what changed,
           - what remains open,
           - what the prior fix exposed.
           Do not re-examine what was already resolved.
           Preserve the same PRECHECK mode unless newly added
           artifacts justify reclassification.
